02 · Why Git is useful

Git lets a project have memory—and parallel futures.

A commit gives a change a durable name and context. A branch lets you try an idea without disturbing another line of work. A remote lets several people exchange those histories.

The real payoff is not merely “backup.” It is the freedom to experiment, compare, explain, review, and recover.

↺RecoverReturn to a known checkpoint.
⑂ExperimentGrow an idea on its own branch.

04 · Read the graph

A commit is a snapshot with a parent.

Most commits point to one earlier commit. Follow those parent links and you get history. The long hexadecimal commit ID is Git's permanent name for that snapshot.

A branch such as main is only a movable label pointing at one commit. HEAD tells Git which branch—or occasionally which commit—you currently have checked out.

Commit C stores a project snapshot, metadata, and the identity of parent B.

The idea to keep
Git stores snapshots. Parent links make history. Branches and HEAD are labels that move.

Branch simulator

A branch is not another folder.

It is a lightweight pointer to a commit. Creating one does not copy the project; committing while it is checked out simply moves that pointer forward.

A conflict is a question, not a catastrophe.

Git can combine different files and different lines automatically. When two histories change the same region differently, Git stops and asks which meaning should survive.

Repository exchange

“Remote” means another repository, not the real one.

Your repository is complete: it has commits, branches, and history. A remote such as origin is another repository whose state you can observe and exchange commits with.

git fetch

Download new commits and update remote-tracking labels such as origin/main. Your branch and files stay put.

git pull

Fetch, then immediately integrate into the current branch—using merge or rebase depending on configuration.

git push

Upload commits and ask the remote branch to move forward. It can refuse if the remote has history you lack.

A calm everyday loop

Inspect before you move history.

A small habit prevents a surprising number of Git emergencies: ask where you are, make one coherent change, inspect it, then exchange history deliberately.

Later lessons can build on this picture to explain undoing with restore, reset, and revert; pausing with stash; naming releases with tags; and recovering with the reflog.
