01 · Start with an analogy

A packet is a little like a letter.

You put a message inside, write a destination on the outside, and hand it to a system that knows the next useful direction—not necessarily the whole route.

The analogy is imperfect, and that is useful. Unlike a postal letter, a large digital message is split into many packets. They may take different routes, arrive out of order, or not arrive at all.

03 · Notice the promises

The internet layer is deliberately modest.

IP tries to move each packet toward its destination. It does not promise delivery, order, or a fixed travel time. That small contract keeps the network flexible.

✓

Best-effort forwardingUse the destination address to choose a next hop.

×

Guaranteed arrivalCongestion, failures, or full queues can drop packets.

×

Guaranteed orderPackets can follow different paths and arrive rearranged.

The idea to keep
IP moves packets. Transport protocols decide what to do when reality gets messy.

Protocol shelf

Same packets. Different promises.

UDP, TCP, and QUIC all sit above IP, but they expose very different contracts to an application.

UDPDatagrams

Send it and keep moving.

UDP adds ports and a checksum, then leaves delivery strategy to the application. There is no built-in handshake, retransmission, or ordering.

Low overheadMessage boundariesApplication-controlled

Useful when fresh data matters more than recovering every old packet, or when a protocol wants to build its own behavior.

TCPByte stream

Deliver every byte, in order.

TCP establishes a connection, detects missing data, retransmits it, and presents one ordered byte stream to the application.

ReliableOrderedCongestion controlled

The trade-off a missing segment can hold back later bytes in the same stream until the gap is repaired.

QUICStreams over UDP

Reliability, with independent lanes.

QUIC builds secure, reliable connections in user space over UDP. Multiple streams share one connection without one stream's loss blocking the others.

TLS built inMultiplexedConnection migration

Why UDP? it gives QUIC a widely deployable packet substrate while QUIC defines the connection behavior above it.

Field kit

Ask the network one precise question.

A tool is easiest to remember when it is attached to the question it answers.

01

ping

“Can I reach that host, and roughly how long is the round trip?”

$ ping example.com

Uses ICMP echo messages. A failed ping is a clue, not proof that the host is down—networks can filter ICMP.

02

traceroute

“Which visible hops does traffic cross on the way there?”

$ traceroute example.com

Changes the IP hop limit to provoke responses along the path. Some hops stay silent, so the result is a partial view.

03

tcpdump

“Which packets are crossing this interface right now?”

$ tcpdump -ni any port 443

A compact command-line packet capture tool. Filters keep a busy machine from becoming a wall of packets.

04

Wireshark

“What is inside this exchange, field by field?”

filter tcp.port == 443

Turns captures into a layered, searchable view of frames, packets, streams, timing, retransmissions, and more.

05

curl

“What did the application protocol negotiate and return?”

$ curl -v https://example.com

Verbose mode exposes DNS, connection setup, TLS negotiation, request headers, and response headers in one trace.

One subtle connection

HTTP/3 uses QUIC, and QUIC uses UDP, but QUIC is not “unreliable HTTP.” QUIC implements reliability, security, congestion control, and streams above UDP's small datagram interface.
